# E2E統合テストチェックリスト(TASK-32)

対象:ステージング環境(`docs/vercel-deploy.md`参照)。決済はStripe/PayPalとも**テストモード**で実施する。

凡例:🤖 = Playwright自動化済み(`e2e/`、`npm run test:e2e`で実行可能)。それ以外は手動確認。

## 1. 国内購入(ja) 🤖(主要経路のみ)

1. `/ja` にアクセスし、トップページが表示される
2. 「商品一覧を見る」→商品を1つ選択
3. ペン機種を選択(受付休止中の機種はグレーアウトし選択不可であることを確認)
4. オプショングループがあれば選択(必須マークの項目は未選択だと「次へ」が押せないことを確認)
5. 「ご要望・詳細をご記入ください」が必須になるオプション(requires_note)を選んだ場合、
   未入力だと進めないことを確認
6. 「ご選択内容の確認」で合計金額が正しいこと(基本価格+オプション差分)を確認
7. 「カートに入れる」→「カートに追加しました。」表示、「カートを見る」導線が出る
8. カートページで数量変更・削除が正しく反映される(小計が再計算される)
9. 「レジに進む」→お届け先情報を入力(氏名・メール・電話・郵便番号・住所必須)
10. 送料が郵便番号に応じて表示される(国内一律送料)
11. 支払い方法「クレジットカード(Stripe)」を選択→「お支払いへ進む」
12. Stripeのホスト型決済ページに遷移する
13. テストカード(`4242 4242 4242 4242`、任意の未来の有効期限、任意のCVC)で決済
14. 決済成功後、`/ja/checkout/complete`にリダイレクトされ、「ご注文ありがとうございました」
    と注文番号が表示される
15. **(手動)** 登録したメールアドレス宛に注文確認メールが届く(Resend経由、日本語)
16. **(手動)** 管理画面`/admin/orders`に該当注文が表示され、決済状態が「paid」になっている
17. カートが決済完了後に空になっていることを確認(再度`/ja/cart`にアクセスして確認)
18. 決済をキャンセルした場合(Stripeの戻る操作等)、`/ja/checkout?cancelled=1`に戻り、
    案内文が表示され、カートの中身が維持されていることを確認

## 2. 海外購入(en)

1. `/en` にアクセスし、英語UIで表示される
2. 商品詳細で価格が`price_international`(円建て)で表示される
3. チェックアウトで配送先の国を選択(JP以外)、送料がEMS地域別に表示される
4. 支払い方法「PayPal」を選択→「Proceed to Payment」
5. PayPal Sandboxアカウントでログインし決済承認
6. `/en/checkout/paypal-return`経由で`/en/checkout/complete`にリダイレクトされ、
   英語の注文完了メッセージが表示される
7. **(手動)** 英語の注文確認メールが届く
8. PayPal承認後にブラウザバック等で二重captureが発生しない(`payment_status`が既にpaidなら
   即座に完了ページへリダイレクトされることを確認)
9. capture失敗時(Sandboxでテスト用に拒否カードを使う等):注文はpendingのまま残り、
   `checkout.paypalCaptureFailed`のエラーメッセージが表示される

## 3. 業務一巡 🤖

1. 管理者でログイン(`/admin/login`)
2. 上記1・2で作成した注文が`/admin/queue`のキューに表示される
3. 対象アイテムを選択(樹種別フィルタでの絞り込みも確認)→「バッチ作成」
4. バッチのかんばん画面(`/admin/batches/[id]`)に遷移し、工程1(木取り)から表示される
5. 「次工程へ」→確認モーダル「実行する」で工程が進む(工程2〜6まで繰り返し)
6. 工程7(検品)で全アイテムのチェックボックスをチェックしないと「バッチ完了」が押せないことを確認
7. 全チェック後「バッチ完了」→バッチが完了状態になる
8. 「製作指示書」リンクが別タブでprint用ページを開く
9. `/admin/shipping`(発送プール)で、該当注文が「発送可能」リストに表示される
   (一部アイテムのみ完成の注文は「待ち(部分完成)」に表示され選択不可であることを確認)
10. 対象を選択→「発送バッチ作成」→発送バッチ詳細画面へ遷移
11. 「クリックポストCSV(国内)」「EMS一覧CSV(海外)」のダウンロードボタンで、文字コード
    (Shift-JIS/UTF-8)を切り替えてCSVがダウンロードできることを確認(宛名CSV、作業項目のCSV出力)
12. 各発送先に配送会社・伝票番号(追跡番号)を入力→「登録」
13. 全件登録完了で「全件の伝票番号登録が完了し、発送バッチは発送済みになりました。」が表示される
14. **(手動)** 発送完了メールが顧客に届く(`fn_mark_shipped`関数内で送信される想定)
15. 顧客側マイページ(`/ja/account/orders/[id]`)で進捗が「発送済」に更新され、追跡番号・
    追跡リンクが表示される

## 4. 差戻し・注文キャンセル

1. バッチのかんばん画面で、あるアイテムに「差戻し」を実行(理由を入力)
2. 該当アイテムがキューに戻り、`production_status`が正しく変化することを確認
3. 管理画面から注文をキャンセル(`payment_status: paid→cancelled`)
4. 連動して、該当注文のqueued状態のorder_itemsも`cancelled`になることを確認
   (既にバッチに入っている・発送済のアイテムは対象外であることも確認)
5. キャンセル後、顧客側マイページで「この注文はキャンセルされました。」が表示される

## 5. 受注自動停止

1. 管理画面の設定(`/admin/settings`)で受注状況に関わる閾値・現在の受付状態を確認
2. 生産キューの滞留等が閾値を超えた状態を作る(またはpg_cronの`fn_check_order_acceptance()`を
   手動実行)→ `accepting_orders_global`がfalseになる
3. フロント側トップページの受注状態バナーが「ただいま受注を一時休止しております」表示に切り替わる
4. 商品詳細ページでカスタマイズ完了後も「カートに入れる」が無効化され、
   「ただいま全体の受注を一時休止しております。」の案内が出る
5. 既にカートに入っている商品がある状態で受付休止になった場合、チェックアウト画面で
   ブロック表示になることを確認

## 6. ゲスト購入→会員登録→履歴表示

1. ログインせずに購入(ゲスト購入)を完了する
2. 購入時と同じメールアドレスで`/ja/account/login`から新規登録する
3. 登録確認メールのリンクから認証を完了する(`/ja/account/callback`)
4. マイページ(`/ja/account`)に、ゲスト購入時の注文が自動的にひも付いて表示される
   (`linkAccountAfterAuth()`によるemail一致リンクの確認)
5. 注文詳細(`/ja/account/orders/[id]`)で内容が正しく表示される

## 7. RLS:他人の注文への直接アクセス遮断

1. 顧客Aでログインし、自分の注文詳細URL(`/ja/account/orders/[Aの注文id]`)にアクセスできる
2. 顧客Aのセッションのまま、顧客Bの注文id(推測または既知の値)で
   `/ja/account/orders/[Bの注文id]`に直接アクセスすると、404または一覧に戻る等、
   Bの注文内容が一切表示されないことを確認
3. 未ログイン状態で`/ja/account`にアクセスすると、ログイン画面へリダイレクトされる
4. Supabase側でも、顧客Aのanonキー+セッションで`select * from orders`をSQL的に試みても
   RLS(`customers can read own orders`)により他人の注文行が返らないことを確認
   (開発者ツール等での直接APIコール確認、任意)
