# 第1部 プロジェクト定義

## 1.1 目的
本システムの中核は**販売ページではなく業務管理**である。

1. 受注→生産→発送→納品の一元管理(最優先)
2. BASEステータスと業務実態の乖離解消
3. 販売サイト4つの統合(BASE日/英・Booth・Stores廃止)
4. AI自動化(メール送信、オーダーメイド要件定義)
5. ブランディング刷新

## 1.2 現行システムの根本問題(BASE調査で確認)
- BASEの注文ステータス(入金待ち/未対応/対応中/対応済/キャンセル)は数日発送の物販前提であり、16〜18週間の受注生産を表現できない
- 発送期限制約の回避のため「便宜的に発送済へ変更(実際は未発送)」という実態と乖離した運用を強いられている
- 生産工程・バッチ・納期推定の管理機能が存在せず、すべて手動・記憶頼み

**→ 新システムは「決済ステータス」と「生産ステータス」を完全分離する。**

## 1.3 制約
- 開発リソース：1日4時間 × 約60日(240h)
- 管理者アカウント：1つ(共有、2〜3名利用想定)
- 移行データ：顧客 約1,300件、商品38件、注文履歴

# 第2部 業務プロセス定義(中核)

## 2.1 生産モデル(ヒアリング確定)

| 項目 | 値 |
|------|-----|
| 製作順序 | 受注順を基本に、**樹種ごとにバッチ編成** |
| バッチサイズ | 約20本/サイクル |
| バッチ所要日数 | 約6日 |
| 発送 | まとめて約6件/回(発送バッチ) |
| オーダーメイド | 標準工程+5〜6工程 |

## 2.2 標準品の製作工程(10ステップ・確定)

```
1. 木取り → 2. 穴あけ → 3. 旋盤加工 → 4. 整形 → 5. 研磨
→ 6. 塗装 → 7. 検品 → 8. 箱詰め → 9. ラベリング → 10. 発送
```

- 工程1〜7は**バッチ単位**で進行(樹種別20本)
- 工程8〜10は**発送バッチ単位**で進行(約6件)
- オーダーメイド品は工程3〜5の間に追加工程が入る(採寸反映・試作・フィッティング調整等、+5〜6工程)※追加工程の正式名称は実装時に確定

## 2.3 業務フロー(To-Be)

```
[受注] 注文確定(決済完了)
   ↓ 自動：注文確認メール(仕様確認+現在の待ち期間)
[キュー] 製作アイテムとして生産キューに登録(受注順)
   ↓ 管理者：樹種別の滞留数を見てバッチ編成(約20本)
[バッチ開始] 製作指示書(仕様一覧)を出力
   ↓ 自動：製作開始メール(完成予定時期)
[工程進行] 木取り〜検品をバッチ単位でチェック進行
   ↓
[発送プール] 検品完了アイテムが溜まる
   ↓ 管理者：約6件で発送バッチ編成
[発送] 宛名データ出力 → 発送 → 伝票番号入力
   ↓ 自動：発送完了メール(追跡番号)
[完了]
```

## 2.4 納期推定・受注自動制御

**推定待ち期間(サイト表示・メール記載用)**
```
週間スループット = 20本 ÷ 6日 × 7日 ≒ 23本/週(初期値、実績移動平均で自動補正)
推定待ち週数 = キュー内未完成本数 ÷ 週間スループット + 安全マージン(樹種バッチ待ち考慮)
```

**受注自動停止(確定仕様)**
- 推定待ち期間 > 90日(3ヶ月) → 受注停止フラグ自動ON、フロントに「受注休止中」表示
- 閾値を下回ったら自動再開(管理者への通知付き)
- 手動オーバーライド可(全体/商品別/バリエーション別)

## 2.5 オーダーメイド(フルオーダー)要件定義フロー

現状(ヒアリング確定)：メール2〜3往復
1. 顧客→ ペンを持った写真・動画、主な用途、1日の使用時間、痛み/疲れの部位、ペンだこの位置、好みの軸形状
2. 工房→ 診断結果と形状提案
3. 顧客→ 承諾と要望
4. 工房→ 受注確認と概算納期

**AI自動化設計(Phase 2)**
- 専用申込フォーム：写真/動画アップロード+構造化質問票(上記6項目)
- Claude APIが一次診断ドラフト生成 → **職人がレビュー・修正して送信**(AIの単独判断では送らない)
- 顧客返信の要望をAIが差分整理 → 職人確定 → 受注確定・納期メール自動送信
- 全往復履歴を注文レコードに紐付け保存

# 第3部 機能要件

## 3.1 業務管理システム(開発ボリュームの主体)

### A. 受注管理
- 注文一覧/詳細(検索・フィルタ：期間/ステータス/樹種/機種/国内外)
- **二重ステータス管理**
  - 決済ステータス：未決済 / 決済済 / 返金 / キャンセル
  - 生産ステータス：受付 / キュー待ち / 製作中(工程位置) / 検品済 / 発送準備 / 発送済 / 完了
- 注文メモ、顧客からの要望・メッセージ表示
- BASE注文データのCSVインポート(移行+並行運用期間用)

### B. 生産管理
- **生産キュー画面**：全未着手アイテムを受注順+樹種別滞留数表示
- **バッチ編成**:樹種でグルーピング→チェック選択→バッチ作成(半自動。完全自動化はしない)
- **バッチかんばん**:工程1〜7を列としたバッチカード進行管理
- **製作指示書出力**:バッチ内全アイテムの仕様一覧(機種/軸形状/仕上げ/ボタン/インク染め/利き手/焼印)をPDF出力
- オーダーメイド品は追加工程付き個別カードとして管理
- 工程実績の記録(バッチ開始日/完了日)→スループット自動算出

### C. 発送管理
- 検品完了アイテムの発送プール表示
- 発送バッチ編成(約6件単位)
- **宛名データCSV出力**(クリックポストまとめ申込形式対応)
- 配送方法別処理:クリックポスト(国内)/ EMS(海外)
- 伝票番号入力 → 発送メール自動トリガー

### D. 顧客管理
- 顧客一覧/詳細(購入履歴、キャンセル歴、累計購入額)
- 注文データからの顧客名寄せ(メールアドレス基準)
- メール送受信履歴の紐付け
- BASE顧客データ約1,300件のインポート

### E. メール自動送信(Claude API + Resend)
| トリガー | タイミング | 内容 |
|---------|-----------|------|
| 注文確定 | 決済完了時 | 注文・仕様確認、現在の推定待ち期間 |
| 製作開始 | バッチ開始時 | 製作開始、完成予定時期 |
| 発送完了 | 伝票番号入力時 | 追跡番号、配送業者リンク |
| 遅延通知 | 手動トリガー | お詫び、新予定日 |

- 日英自動判定(注文言語に基づく)
- 送信前プレビュー・編集・承認画面(自動送信のON/OFF設定可)

### F. ダッシュボード
- 本日のサマリー:新規注文/キュー総数/推定待ち週数/受注受付状態
- 工程別仕掛かり数、発送プール数
- 週間スループット推移

## 3.2 販売フロントエンド

### 商品構造(BASE実データ準拠)
```
商品(38種) → バリエーション(対応ペン機種) → オプション(10グループ)
```
- 主要オプション:グリップ形状(14種)/表面仕上げ(4種)/ボタン調整(5種)/インク染め(24色)/利き手/Apple Pencil機種・仕様/焼印
- オプショングループの整理統合を移行時に実施(同名重複の解消)

### 価格体系(確定)
- **国内価格と海外価格は別建て**(二重価格を維持)
  - 理由:国内外の物価差、および長い納品待ち期間中の為替変動(円安)による利益毀損リスクへの対応
  - 商品マスタに国内価格・海外価格の2フィールドを持ち、独立して改定可能
- 送料は価格と別建て表示(B案):国内 ¥185(クリックポスト)/ 海外 EMS地域別 ¥650〜¥1,150
- サイト区分と価格の対応:`/ja/`=国内価格・国内配送のみ、`/en/`=海外価格・海外配送(現行の日英BASE 2店舗体制を踏襲)

### 主要画面
- トップ(ブランドコンセプト:和の美意識×天然素材×リデザイン)
- 商品一覧(フィルタ:機種/シリーズ/樹種/価格)
- 商品詳細+ステップ式カスタマイズUI(機種→形状→仕上げ→ボタン→追加オプション)
- 受注状態表示(受付中/休止中+現在の推定待ち期間)
- カート/チェックアウト(Stripe主+PayPal副)
- 顧客アカウント(注文履歴・製作ステータス確認・追跡)
- オーダーメイド申込フォーム(Phase 2)
- ガイド(ペンの見分けかた/発送について/FAQ)※BASEから移行
- ブログ/特集(Sanity)
- 特商法表記・プライバシーポリシー(日英)

### 多言語
- `/ja/` `/en/` パス分離、ブラウザ言語検出+手動切替

# 第4部 データモデル概要

主要テーブル(詳細ER図はPhase 0で作成):

| テーブル | 内容 |
|---------|------|
| products | 商品マスタ(シリーズ/樹種/国内価格/海外価格/日英名) |
| variations | ペン機種バリエーション(受注可否フラグ) |
| option_groups / option_values | オプション定義(価格差分付き) |
| product_options | 商品↔オプショングループ割当 |
| customers | 顧客(BASE移行+新規) |
| orders | 注文(決済ステータス/言語/送料/合計) |
| order_items | 注文明細=**製作アイテム**(選択仕様のスナップショット/生産ステータス) |
| production_batches | 製作バッチ(樹種/開始日/工程位置) |
| batch_items | バッチ↔製作アイテム |
| shipping_batches | 発送バッチ |
| custom_order_threads | オーダーメイド要件定義の往復履歴 |
| emails | 送信メールログ |
| production_metrics | スループット実績 |

設計原則:
- order_itemsに注文時点の仕様・価格を**スナップショット保存**(マスタ変更の影響を受けない)
- 生産ステータスはorder_items(アイテム単位)に持ち、バッチ経由で一括更新

| データ | 件数 | 方法 |
|--------|------|------|
| 商品・オプション | 38商品+10オプショングループ | BASE商品CSVエクスポート→変換スクリプト+手動整理 |

**並行運用**:新業務管理システム稼働後、英語BASEの新規注文はCSVインポートで取り込み、生産管理は新システムで一本化。フロント公開後にBASEを閉鎖・リダイレクト。

# 第7部 開発フェーズ計画(240h・業務システム優先)

| Phase | 内容 | 工数 | 完了時の価値 |
|-------|------|------|-------------|
| 0 | 詳細設計(ER図/画面設計/API設計)、環境構築 | 30h | — |
| 1 | **業務管理コア**:DB構築、受注管理、生産キュー、バッチかんばん、BASEインポート | 60h | 即日から生産管理に使用可能 |
| 2 | 発送管理、メール自動化(Claude+Resend)、ダッシュボード | 40h | 発送・連絡の自動化 |
| 3 | 販売フロント:商品/カスタマイズUI/カート/Stripe/PayPal/顧客アカウント | 60h | 自社EC稼働 |
| 4 | ブランディングページ、多言語、Sanity、SEO | 30h | フルサイト公開 |
| 5 | データ移行、統合テスト、DNS切替、BASE閉鎖 | 20h | 一元化完了 |

**業務管理を先に作る理由**:フロント完成を待たずにBASE注文をインポートして生産管理を開始でき、2ヶ月の開発期間中も現場が改善される。リスクも分散する。

Phase 2以降の拡張(スコープ外):オーダーメイドAIチャット、配送業者API連携、複数管理者ロール
